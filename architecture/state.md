# Global State Management

The single source of truth for saved data (Presets) and the current workspace configuration. It bridges the gap between the stateless EffectEngine and the Vue UI.

## Pinia (`engine-store.ts`)
- Hosts the true global state of the workspace: `sceneNodes`, `flatFixtures`, and `selectedIds`.
- Instantiates the `EffectEngine` and controls the `dmxBuffer`, running the `requestAnimationFrame` loop globally so the math never stops when pages change.
- Tracks the central repository of `savedPresets` and `selectedPresetId`.
- Used extensively by `use-presets.ts` to manage loading, saving, and diffing against the DMX output states without mutating raw Vue component variables.

### Why do Presets store Fixture References (`fixtureIds`)?
A Preset in Pixelmapper goes beyond traditional flat "global palettes." It functions as an explicitly mapped snapshot:
- **Selective Application**: By saving an array of `fixtureIds` within each category block of a `Preset`, the system knows *exactly* which fixtures this preset was authored for. 
- **Modularity**: A user can create a "Red Color" preset exclusively for their Wash lights without accidentally blowing out the colors on their Spot lights when the preset is triggered.
- **Diffing & Tracking**: When the user edits a fader, the UI needs to know which preset is currently active and whether it has unsaved changes. The engine scans the live programmer state against the preset's snapshots (filtered by these `fixtureIds`). If a user selects a fixture *not* referenced in the preset, changing its channels won't artificially flag the preset as "modified."

## Reactivity Strategy (Data-Oriented Design)
Pixelmapper handles hundreds of mathematical DMX calculations per frame. If Vue deeply tracked every property of 50+ fixtures (positions, arrays of channels, step values, config objects), the rendering would crawl to a halt as massive dependency graphs are constantly torn down and rebuilt.

To solve this, Pixelmapper uses a strict Data-Oriented Design (DoD) reactivity approach:

1. **The Engine is `markRaw()`**: The `EffectEngine` and its `Uint8Array` / `Float32Array` buffers exist entirely outside of Vue's view. Math happens directly in contiguous memory.
2. **`sceneNodes` is `shallowRef`**: Fixtures are plain JavaScript objects. Adding or removing a fixture replaces or triggers the array reference, updating the UI. However, modifying a deep property like a fader value (`channel.chaserConfig.stepValues[0]`) does **not** trigger Vue's observers.
3. **Canvas Detachment**: The 60fps canvas loop does not use `watchEffect`. Instead, it manually watches for shallow changes to `camera` and `fixtureColors` (a Map generated by the engine loop 60 times a second), ensuring high-frequency rendering never enters Vue's tracking system.

### The History Ticker Pattern
Because fixtures are plain objects inside a `shallowRef`, how does the UI know when a deeply nested property changes (e.g., to show an "Unsaved Preset" dot)?

The system relies on a **History Ticker**:
- `useHistory()` exposes a reactive `version` ref.
- Every user interaction that mutates fixture data (like dragging a fader or checking a box) is wrapped in a `Command`.
- When a `Command` executes, undoes, or redoes, `version.value++` is triggered.
- UI components (like properties sidebars or preset lists) explicitly depend on `history.version.value` inside their `computed` functions.
- This forces Vue to re-evaluate the computation precisely *when* the user performs an action, bypassing the need for deep proxies entirely.
